# 日志存储升级说明

## 版本信息
- **升级版本**: V2
- **升级日期**: 2024-01-15
- **影响范围**: 日志存储机制

## 升级内容

### 核心变更
将日志存储方式从**数据库存储**改为**文件系统存储**，解决大日志导致的数据库性能问题。

### 设计方案

#### 1. 日志文件命名规则
```
格式：taskId_executionCount_startTime.log
示例：abc123_1_20240115143020.log
     abc123_2_20240115150130.log
```

#### 2. 目录结构
```
logs/
  tasks/
    2024/
      01/
        abc123_1_20240115143020.log
        abc123_2_20240115150130.log
        def456_1_20240115144500.log
      02/
        ...
```

#### 3. 数据流程
```
Agent执行任务 -> 上报日志 -> 服务端写入文件 -> 查询时从文件读取
```

## 数据库变更

### Task表新增字段
```sql
ALTER TABLE tasks ADD COLUMN execution_count INT DEFAULT 0;
ALTER TABLE tasks ADD COLUMN log_file_path VARCHAR(500);
```

### TaskLog表处理
- **保留表结构**，但不再使用
- 待系统稳定后可选择清理或删除

## 代码变更

### 后端修改

#### 1. Task.java
- 新增字段：`executionCount`（执行次数）
- 新增字段：`logFilePath`（日志文件路径）

#### 2. TaskService.java
- `ackTask()`：生成日志文件路径，executionCount累加
- `appendLog()`：改为写入文件而非数据库
- 新增方法：`generateLogFilePath()`
- 新增方法：`writeLogToFile()`

#### 3. WebController.java
- `getTaskLogs()`：改为从文件读取，支持分页
- 新增接口：`downloadTaskLog()` - 下载日志文件
- 新增方法：`readLogFile()` - 统一的文件读取逻辑

### 前端修改

#### 1. tasks.js
- 新增状态：`taskLogContent`、`taskLogTotalLines`、`taskLogHasMore`
- 修改方法：`refreshLogs()` - 适配新的API响应格式
- 新增方法：`downloadTaskLog()` - 下载日志文件

#### 2. tasks.html
- 日志对话框增加：下载按钮、行数显示

### 配置文件
```yaml
lightscript:
  log:
    storage:
      path: logs/tasks  # 日志存储根目录
    retention:
      days: 90  # 日志保留天数
```

## 升级优势

| 对比项 | 升级前（数据库） | 升级后（文件） |
|--------|----------------|---------------|
| **写入性能** | 🔴 数据库INSERT，慢 | ✅ 文件追加，快 |
| **容量限制** | 🔴 受数据库限制 | ✅ 受磁盘限制（更大）|
| **超大日志** | 🔴 可能超过字段限制 | ✅ 无限制 |
| **查询性能** | ✅ 数据库查询快 | 🟡 文件读取稍慢 |
| **备份迁移** | 🟡 随数据库备份 | ✅ 单独文件，灵活 |
| **清理策略** | 🔴 需要DELETE | ✅ 直接删除文件 |

## 使用说明

### 查看日志
1. 在任务列表点击"日志"按钮
2. 系统自动从文件加载日志（前5000行）
3. 显示总行数和是否有更多内容

### 下载日志
1. 在日志对话框点击"下载日志"按钮
2. 浏览器自动下载完整的日志文件
3. 文件名格式：`taskId_executionCount_startTime.log`

### 日志文件管理
- **存储位置**: `logs/tasks/YYYY/MM/`
- **命名格式**: `taskId_executionCount_startTime.log`
- **保留期限**: 90天（可配置）
- **清理方式**: 定期删除过期的月份目录

## 注意事项

### 1. 数据迁移
- 升级后首次启动会自动执行数据库迁移
- 已有任务的日志仍在数据库中（暂不清理）
- 新任务的日志将存储到文件系统

### 2. 磁盘空间
- 日志文件会占用磁盘空间
- 建议定期清理过期日志
- 监控磁盘使用情况

### 3. 备份策略
- 日志文件需单独备份
- 可使用文件同步工具定期备份`logs/tasks/`目录
- 备份时按月份目录选择性备份

### 4. 服务器迁移
- 迁移时需要同时迁移`logs/tasks/`目录
- 或者只迁移数据库，历史日志丢失（仅影响查看）

## 回滚方案

如果需要回滚到旧版本：

1. **数据库回滚**
```sql
ALTER TABLE tasks DROP COLUMN execution_count;
ALTER TABLE tasks DROP COLUMN log_file_path;
```

2. **代码回滚**
- 恢复之前的代码版本
- TaskLog表仍然保留，可继续使用

3. **数据处理**
- 文件系统中的日志可保留作为备份
- 或者手动导入到TaskLog表

## 技术支持

如遇到问题，请检查：
1. 日志文件路径配置是否正确
2. 服务器是否有写入权限
3. 磁盘空间是否充足
4. 日志文件是否存在
